# GSD Orchestrator 아키텍처

> Telegram/Slack ↔ Claude Code ↔ GSD 프레임워크를 파일 기반 IPC로 연결하는 무인 자동화 오케스트레이터.
> 단일 Python 프로세스로 동작. cron 불필요.

## 컴포넌트 구성

### 패키지: `src/gsd_orchestrator/`

```
src/gsd_orchestrator/
├── __main__.py              ← 진입점 (main 함수, 로그 설정)
├── config.py                ← config.yaml + .env 로딩 (채널/브로드캐스트 설정 포함)
├── api.py                   ← 외부 연동 인터페이스 (ChannelSender)
├── orchestrator.py          ← 메시지 핸들링 + 오케스트레이션 (+ on_result 콜백 연결점)
├── channels/                ← 채널 어댑터 패키지
│   ├── base.py              ← ChannelAdapter ABC (분할 발송 캡슐화)
│   ├── manager.py           ← ChannelManager (브로드캐스트, targets 빌드)
│   ├── telegram.py          ← TelegramAdapter (슬래시 커맨드 포함)
│   └── slack.py             ← SlackAdapter (지연 임포트, Socket Mode)
├── attachment_handler.py     ← 첨부파일 처리 (화이트리스트, 다운로드, 텍스트 추출, 메타데이터)
├── text_cleaner.py           ← 추출 텍스트 정제 (공백/헤더/푸터/페이지번호 제거, PDF 특화)
├── inbox_writer.py          ← 메시지 → JSON 파일 (원자적 쓰기, source 객체, attachments)
├── inbox_processor.py       ← inbox 폴링 → 분류 → Claude/GSD → outbox 조립 + 작업 큐 실행
├── outbox_sender.py         ← outbox 폴링 → 멀티채널 발송 (retry, snippet)
└── archiver.py              ← sent/ → archive/ 이동 + 만료 삭제
```

### 채널 어댑터 계층

```
ChannelAdapter (ABC)
├── channel_type: str            # "telegram" | "slack"
├── max_message_length: int      # 4096 | 4000
├── start(on_message)            # 수신 시작 + 봇 메시지 skip
├── stop()                       # 정지
├── send_message(ch_id, text)    # 내부 분할 발송 캡슐화
└── get_channel_id()             # 단일 채널 ID

ChannelManager
├── send_to(type, id, text)      # 특정 채널 발송
├── broadcast_all(text)          # 모든 채널 발송
├── build_broadcast_targets()    # origin(is_origin=true) + 상대 채널
├── start_all() / stop_all()     # 어댑터 일괄 제어
└── get_all_channels()           # [(type, id), ...]
```

### orchestrator.py — 메시지 핸들링

모든 채널 메시지의 공통 콜백 `_on_channel_message(source, text, mode)`:
1. inbox에 source 객체 포함하여 저장
2. `[채널][사용자][키워드] 작업중입니다.` 모든 채널에 broadcast
3. GSD 블로킹 상태이면 `mode="gsd-resume"`로 저장

### 슬래시 커맨드 (TelegramAdapter)

| 핸들러 | 트리거 | 동작 |
|--------|--------|------|
| `_on_message` | 일반 텍스트 | 콜백 → inbox `mode="default"` |
| `_on_gsd` | `/gsd <작업>` | 콜백 → inbox `mode="gsd"` |
| `_on_reset` | `/reset` | 다음 요청 시 새 세션 |
| `_on_status` | `/status` | 쿨다운, 블로킹, 토큰 사용량 |
| `_on_resume` | `/resume` | 쿨다운 즉시 해제 |
| `_on_document` | 파일 첨부 | 화이트리스트 검증 → 텍스트 추출 → 정제 → 확인 요청 |

### 백그라운드 태스크 (Orchestrator)

| 태스크 | 간격 | 동작 |
|--------|------|------|
| `inbox_processor.run()` | 10초 | inbox → 분류 → Claude/GSD → outbox |
| `outbox_sender.run()` | 3초 | outbox → 멀티채널 발송 → sent/ |
| `_inbox_stale_check()` | 60초 | inbox 5분 이상 미처리 시 경고 |
| `_run_archiver()` | 1시간 | sent/ → archive/ → 만료 삭제 |

## 멀티채널 아키텍처

### 채널 구성

- Telegram 1개, Slack 1개 — 각각 선택적, 최소 1개 필수
- 봇 자신의 메시지 skip (Telegram: `is_bot`, Slack: `bot_id`)

### 응답 포맷

모든 응답에 `[채널][사용자명][요청키워드]` 헤더:

```
[Telegram][김철수][결제모듈구현] 작업중입니다.        ← 수신 확인
[Telegram][김철수][결제모듈구현] 처리결과              ← 완료 응답
[시스템] 세션이 자동 리셋되었습니다.                    ← 시스템 알림
```

### 크로스채널 브로드캐스트

- 수신 확인: 모든 채널에 동일 메시지
- 처리 완료: origin 채널은 전체 응답, 상대 채널은 500자 요약본(snippet_length 설정)
- 시스템 알림: 모든 채널에 동일

### outbox targets 구조

```json
{
  "targets": [
    {"channel_type": "telegram", "channel_id": "123", "is_origin": true},
    {"channel_type": "slack", "channel_id": "C456", "is_origin": false}
  ],
  "retry_count": 0
}
```

- `is_origin=true`: 전체 응답
- `is_origin=false`: snippet_length 이내로 잘림 + "[전체 내용은 origin에서 확인하세요]"
- `retry_count`: 부분 실패 재시도 횟수, MAX_OUTBOX_RETRIES(3) 초과 시 error/로 이관

## 메시지 JSON 포맷

파일명: `{YYYYMMDD}_{HHMMSS}_{uuid8}.json`

```json
{
  "id": "uuid",
  "source": {
    "channel_type": "telegram",
    "channel_id": "123456789",
    "user_id": "U789",
    "user_name": "김철수",
    "message_id": 42,
    "thread_ts": null
  },
  "chat_id": "123456789",
  "message_id": 42,
  "keyword": "결제모듈구현",
  "mode": "default | gsd | gsd-resume",
  "request": {
    "text": "결제 모듈 구현해줘",
    "timestamp": "2026-03-19T14:30:52+09:00"
  },
  "response": null
}
```

- `source`: 채널/사용자 정보 (멀티채널용)
- `chat_id`, `message_id`: 하위 호환 필드
- `thread_ts`: Slack 스레드 식별 (향후 확장용)

## 메시지 상태 전이

```
messages/inbox/xxx.json                요청 생성 (response: null)
    │
    ├──▶ inbox/xxx.json.processing     처리 중 (lock)
    │        │
    │        ├──▶ outbox/xxx.json      응답 + targets 추가
    │        │        │
    │        │        ├──▶ xxx.json.sending    발송 중 (중복 방지)
    │        │        │        │
    │        │        │        ├──▶ sent/xxx.json      발송 완료
    │        │        │        │        │
    │        │        │        │        └──▶ archive/YYYY-MM-DD/   보관
    │        │        │        │
    │        │        │        └──▶ (부분 실패: retry_count++ → outbox 복원)
    │        │        │
    │        │        └──▶ error/xxx.json   retry_count >= 3 → 격리
    │        │
    │        └──▶ inbox/xxx.json       처리 실패 시 복원
    │
    └──▶ error/xxx.json     파일 3회 연속 실패 → 격리
```

## Two-Track 라우팅

```
사용자 메시지 수신 (어느 채널이든)
    │
    ├─ /gsd 명령어 → mode="gsd" (.gsd-active 삭제 → 새 GSD 세션)
    │
    ├─ .blocked 존재 → mode="gsd-resume" (GSD 블로킹 응답)
    │
    ├─ .gsd-active 존재 (타임아웃 미초과) → mode="gsd-resume" (세션 이어가기)
    │
    └─ 없음 → mode="default"
         │
         └─ 경량 모델 분류 (--no-session-persistence)
              │
              ├─ "gsd" → GSD Track (claude -p "/gsd:do ...")
              │
              └─ "simple" → Simple Track (claude -p --continue)
```

### GSD 세션 유지

GSD 트랙 처리가 성공하면 `.gsd-active` 런타임 파일(PID 기록)을 생성하여 세션 활성 상태를 표시한다. 다음 메시지가 도착하면 `gsd-resume` 모드로 이전 세션을 `--continue`로 이어간다.

```
GSD 처리 완료 → .gsd-active 생성 (PID + timestamp)
    │
    ├── 다음 메시지 (동일 PID) → --continue (세션 유지)
    │
    ├── 다음 메시지 (PID 불일치, 프로세스 재시작) → sent/ 이력 복원
    │     └── 최근 대화 이력을 프롬프트에 주입하여 맥락 복원
    │
    ├── 30분 타임아웃 → .gsd-active 삭제 → 새 요청으로 처리
    │
    └── /gsd 명령어 or /reset → .gsd-active 삭제 → 새 세션
```

### GSD 작업 큐

복잡한 작업(10분 초과 예상)은 계획서 → 단위별 순차 실행으로 분할한다. 타임아웃 시에도 완료된 단위의 결과가 보존된다.

```
사용자 요청
    │
    ├── Claude 판단: 단순 → 즉시 실행 (기존과 동일)
    │
    └── Claude 판단: 복잡 → .gsd-plan.md 작성 → 사용자에게 계획 보고
          │
          └── 사용자 확인 → .gsd-plan.md 파싱 → workqueue/ 생성
                │
                ├── Unit 1: claude -p → 완료 체크 → 결과 보고
                ├── Unit 2: claude -p → 완료 체크 → 결과 보고
                └── 전체 완료 → 최종 보고 + 정리
```

- `messages/plan/.gsd-plan.md`: 계획서, 단위별 `- [ ]`/`- [x]` 체크리스트
- `messages/workqueue/`: 단위별 JSON 파일, inbox와 동일한 폴링으로 순차 처리
- Unit 실패 3회 → skip + 다음 Unit 진행, 알림 발송
- 프로세스 재시작 시 `workqueue/` 파일이 남아있어 자동 재개

### 소진 감지 및 쿨다운

```
claude -p 실패
  → 파일별 실패 +1 (3회 → error/ 격리)
  → 전역 실패 +1 (5회 → 쿨다운)
    → /tmp/gsd-orchestrator.cooldown (재개 시각 기록)
    → 채널 알림 + /resume 명령으로 즉시 해제
    → 5시간 이상 쿨다운 시 1회 경고
```

### 세션 관리

```
claude -p --continue  ← 세션 이어가기 (Simple Track)
  ├── /reset → 다음 호출 시 --continue 제거
  └── max_session_turns(20) 초과 → 자동 리셋 + 알림

분류 호출: --no-session-persistence (세션 오염 방지)
GSD Track: 자체 fresh context
```

## 런타임 파일

프로젝트 경로의 MD5 해시(8자)로 인스턴스별 격리. `{id}` = 인스턴스 해시.

```
/tmp/gsd-orchestrator-{id}.pid              ← 프로세스 ID
/tmp/gsd-orchestrator-{id}.cooldown         ← 쿨다운 재개 시각
/tmp/gsd-orchestrator-{id}.cooldown-alerted ← 5시간 경고 발송 여부
/tmp/gsd-orchestrator-{id}.reset            ← /reset 신호 파일
/tmp/gsd-orchestrator-{id}.turncount        ← 세션 턴 카운트
/tmp/gsd-orchestrator-{id}.failcount        ← 연속 실패 카운트
/tmp/gsd-orchestrator-{id}.blocked          ← GSD 블로킹 상태
/tmp/gsd-orchestrator-{id}.active           ← 재개 감지용
/tmp/gsd-orchestrator-{id}.token-usage      ← 누적 토큰 사용량
```

여러 인스턴스가 같은 머신에서 동시 실행되어도 런타임 파일이 충돌하지 않습니다.

## 외부 연동 인터페이스

외부 애플리케이션과 두 가지 모드로 연동할 수 있습니다.

### Library 모드 (같은 프로세스)

패키지 형태로 호스트 앱에 import하여 Python API로 연동합니다. 인터페이스 미등록 시 단독 실행과 동일하게 동작합니다.

1. **`on_result(callback)` — 처리 결과 콜백 (Orchestrator → Host App)**
   - Claude/GSD가 요청 처리를 마치면 결과를 콜백으로 전달합니다.
   - 콜백 인자: `(source: dict, request_text: str, response_text: str, status: str)`
   - `status`: `"success"` | `"error"` | `"blocked"` (성공/실패/GSD 블로킹)

2. **`ChannelSender` — 메시지 전송 요청 (Host App → Orchestrator)**
   - `orchestrator.channel_sender`로 접근합니다.
   - `send(text, channel_type?)`: 즉시 발송 (네트워크 실패 시 False)
   - `enqueue(text, channel_type?)`: outbox 파일 큐 경유, 최대 3회 재시도 보장

진입점: `run()`(블로킹) 또는 `start()`/`stop()`(비동기, 호스트 앱의 기존 이벤트 루프에 추가 가능).

### 독립 프로세스 모드 (파일 기반)

GSD-Orchestrator가 별도 프로세스로 실행 중일 때, 파일 시스템을 통해 연동합니다. 별도 패키지 설치 불필요.

- **요청**: `messages/inbox/`에 JSON 파일을 원자적으로 작성 (`.tmp` → `rename`)
- **결과**: `messages/sent/`에서 처리 완료된 JSON 파일을 폴링
- **에러**: `messages/error/`에 격리된 파일을 `inbox/`로 이동하여 재처리

상세 사용법은 [`docs/외부-연동-가이드.md`](외부-연동-가이드.md) 참조.

## 첨부파일 처리 흐름

텔레그램/슬랙에서 파일이 첨부되면 다음 흐름으로 처리된다.

```
사용자 파일 첨부 (Telegram Document / Slack file_shared)
    │
    ├── validate_file() → 화이트리스트(txt, md, pdf) + 크기(1MB) 체크
    │     └── 거부 시 "txt, md, pdf 파일만 지원합니다" 즉시 회신
    │
    ├── download_file_telegram() / download_file_slack()
    │     └── messages/attachments/ 임시 저장
    │
    ├── extract_text() → txt/md 직접 읽기, pdf는 pdfminer.six
    │     ├── PDF 암호화 → "[오류] 이 PDF는 보호되어 열 수 없습니다"
    │     ├── PDF 이미지 전용 → "[오류] 이미지로 구성되어 텍스트 추출이 불가합니다"
    │     └── PDF 구문 오류 → "[오류] PDF 파일 형식이 올바르지 않습니다"
    │
    ├── clean_text() → 정제 (공백, 헤더/푸터, 페이지번호, 하이픈 줄바꿈 등)
    │
    ├── 사용자에게 정제된 텍스트 확인 요청
    │
    ├── inbox 메시지에 attachments[] 메타데이터 + extracted_text 추가
    │
    └── cleanup_temp_file() → 임시 파일 삭제
```

### 메시지 JSON 확장 필드

```json
{
  "request": {
    "text": "이 문서 분석해줘",
    "attachments": [
      {
        "filename": "report.pdf",
        "extension": "pdf",
        "uploaded_at": "2026-03-24T10:30:00+09:00",
        "size_bytes": 524288
      }
    ],
    "extracted_text": "[첨부파일: report.pdf]\n정제된 텍스트 내용..."
  }
}
```

## 설계 원칙

1. **단일 프로세스** — cron/데몬 분리 없이 하나의 Python 프로세스가 모든 기능 수행
2. **완전 파일 기반** — DB 없음. 파일 이동 = 상태 전이. `ls`와 `mv`로 디버깅/복구
3. **원자적 쓰기** — `.tmp` → `rename`. 불완전한 파일이 읽히지 않음
4. **중복 방지** — inbox는 `.processing`, outbox는 `.sending`으로 원자적 lock
5. **발송 일원화** — 모든 채널 발송은 outbox/ 경유. 시스템 알림도 outbox에 JSON 작성
6. **세션 오염 방지** — 분류 호출은 `--no-session-persistence`로 본 세션 보존
7. **무인 자동 운영** — 토큰 소진 → 자동 cooldown → `/resume`으로 즉시 재개
8. **채널 추상화** — ChannelAdapter ABC로 Telegram/Slack 외 채널 확장 가능
9. **봇 메시지 skip** — 어댑터 레벨에서 봇 메시지를 원천 차단, 무한 루프 방지

## 장애 복구 테스트

장애-복구-가이드.md에 정의된 장애 시나리오를 실제로 발생시켜 자동 복구 메커니즘과 알림 발송이 올바르게 동작하는지 검증한다. 테스트는 Mock을 최소화하고 실제 파일 I/O로 장애 상황을 재현한다.

### 테스트 전략

- `_run_claude`만 AsyncMock으로 대체 (외부 프로세스 의존 제거)
- `FakeAdapter` 패턴으로 채널 발송 추적 (`sent_messages` 리스트)
- 알림 파일(`outbox/*_system-alert.json`) 생성 여부로 알림 발송 검증

### Mock 구조

```
InboxProcessor
  ├── Config ← 실제 Config.load() (config.yaml 자동 생성)
  ├── ChannelManager ← FakeAdapter (sent_messages 추적)
  └── _run_claude ← AsyncMock (성공/실패/타임아웃 시뮬레이션)

OutboxSender
  ├── ChannelManager ← FakeAdapter / HtmlFailAdapter
  └── 파일 I/O ← 실제 tmp_path

Orchestrator._inbox_stale_check
  ├── ChannelManager ← FakeAdapter
  └── inbox 파일 mtime ← os.utime()으로 조작
```
