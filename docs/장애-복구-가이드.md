# GSD Orchestrator 장애 복구 가이드

GSD Orchestrator는 파일 기반 상태 관리로 대부분의 장애에서 자동 복구가 가능합니다. 이 문서는 장애 유형을 분류하고 각 유형별 자동/수동 복구 절차를 정의합니다.

---

## 자동 복구 흐름

프로세스 재기동 시 다음 순서로 자동 복구가 진행됩니다.

```
프로세스 재기동 (./start.sh)
    │
    ├── start.sh: cooldown, failcount, lock 파일 삭제
    │
    ├── InboxProcessor._recover_stale_processing()
    │   └── .processing 파일 → failcount 확인 → 복원 또는 error/ 격리
    │
    ├── OutboxSender._recover_stale_sending()
    │   └── .sending 파일 → outbox/로 무조건 복원
    │
    └── 정상 폴링 루프 시작
        ├── inbox 폴링 (inbox_check_interval, 기본 10초)
        ├── outbox 폴링 (outbox_interval, 기본 3초)
        ├── inbox stale check (60초) → 5분 미처리 경고 브로드캐스트
        └── archiver (1시간) → sent/ → archive/ → 만료 삭제
```

---

## 수동 복구 명령어 요약

| 상황 | 명령어 |
|------|--------|
| 프로세스 중단 | `./start.sh` |
| 프로세스 재시작 | `./restart.sh` |
| 쿨다운 해제 | 채널에서 `/resume` |
| 격리 메시지 재처리 | `mv messages/error/파일.json messages/inbox/` |
| 격리 메시지 재발송 | `mv messages/error/파일.json messages/outbox/` |
| 토큰 사용량 초기화 | `rm /tmp/gsd-orchestrator-*.token-usage` |
| 대기 상태 확인 | `ls messages/inbox/ messages/outbox/ messages/error/` |

## 카테고리 A: 프로세스 장애

### A-1. 프로세스 비정상 종료

**원인:** OOM, SIGKILL, 시스템 재부팅

**영향:** 진행 중 작업 중단. `.processing`/`.sending` 상태의 파일이 잔류.

**자동 복구:**
- 재기동 시 `_recover_stale_processing()`이 `.processing` 파일을 감지
- failcount 확인 후 3회 미만이면 inbox/로 복원, 3회 이상이면 error/로 격리
- `_recover_stale_sending()`이 `.sending` 파일을 outbox/로 무조건 복원

**수동 대응:**
```bash
./start.sh
```

**코드 참조:**
- `inbox_processor.py:97-136` — `_recover_stale_processing()`
- `outbox_sender.py:27-35` — `_recover_stale_sending()`
- `start.sh:57-60` — cooldown/failcount/lock 파일 정리

### A-2. 프로세스 좀비화

**원인:** 자식 프로세스(claude CLI)가 종료되지 않아 메인 프로세스가 응답 불가

**영향:** PID 파일은 존재하지만 실제 메시지 처리 불가

**자동 복구:** 없음

**수동 대응:**
```bash
./stop.sh       # kill_tree()로 자식 프로세스까지 재귀 종료
./start.sh
```

**코드 참조:**
- `stop.sh:9-20` — `kill_tree()` 재귀 프로세스 종료

### A-3. 터미널 종료로 프로세스 중단

**원인:** 포그라운드 실행 시 터미널 세션 종료

**영향:** 전체 서비스 중단

**자동 복구:** `start.sh`가 `nohup`으로 데몬화하여 발생하지 않음

**수동 대응:**
```bash
./start.sh      # nohup 데몬으로 시작
```

**코드 참조:**
- `start.sh:67` — `nohup python -m gsd_orchestrator > /dev/null 2>&1 &`

## 카테고리 B: Claude 실행 장애

### B-1. Claude 타임아웃

**원인:** 복잡한 작업, 네트워크 지연

**영향:** 해당 요청 실패 처리 (`_run_claude()`가 `None` 반환)

**자동 복구:**
- `asyncio.wait_for` 타임아웃 → `proc.kill()` → `None` 반환
- `_handle_failure()` → failcount++ → 3회 시 error/ 격리
- 전역 failcount 5회 시 쿨다운 진입

**수동 대응:**
```yaml
# config.yaml — timeout 값 상향
claude:
  timeout: 600    # 기본 300초
```

**코드 참조:**
- `inbox_processor.py:400-412` — 타임아웃 처리
- `inbox_processor.py:565-604` — `_handle_failure()`

### B-2. Claude 응답 파싱 실패

**원인:** Claude CLI가 비정상 JSON 출력

**영향:** 해당 요청 실패 처리

**자동 복구:**
- `json.JSONDecodeError` 캐치 → `None` 반환
- `_handle_failure()` → failcount++ → 3회 시 error/ 격리

**수동 대응:**
```bash
# error/ 파일 확인 후 재처리
cat messages/error/파일.json | python -m json.tool
mv messages/error/파일.json messages/inbox/
```

**코드 참조:**
- `inbox_processor.py:422-423` — JSON 파싱 실패 처리

### B-3. Claude CLI 미설치/경로 오류

**원인:** PATH 설정 오류, claude CLI 미설치

**영향:** 모든 요청 실패

**자동 복구:**
- `_run_claude()`에서 `_EXTRA_PATHS`로 PATH 보강 (`.local/bin`, `/opt/homebrew/bin`, `/usr/local/bin`)
- 실행 실패 시 전역 failcount → 5회 시 쿨다운 + 채널 알림

**수동 대응:**
```bash
# claude CLI 설치 확인
which claude
# 미설치 시
npm install -g @anthropic-ai/claude-code
```

**코드 참조:**
- `inbox_processor.py:22-29` — PATH 보강
- `inbox_processor.py:590-604` — 전역 실패 → 쿨다운

### B-4. 토큰 소진 (구독 한도)

**원인:** 일일 사용량 초과

**영향:** 모든 요청 실패

**자동 복구:**
- 전역 failcount 5회 → 쿨다운 진입 (`cooldown_retry_minutes`, 기본 10분)
- 쿨다운 5시간 이상 지속 시 추가 경고 알림
- 쿨다운 만료 시 자동 재시도

**수동 대응:**
```
채널에서 /resume     # 쿨다운 즉시 해제
```

**코드 참조:**
- `inbox_processor.py:648-671` — `_is_cooldown()`
- `inbox_processor.py:593-604` — 쿨다운 진입
- `telegram.py:196-213` — `/resume` 커맨드

### B-5. Claude 작업 중 프로세스 중단

**원인:** A-1과 동시 발생

**영향:** Claude 결과 유실, 처음부터 재처리

**자동 복구:**
- `.processing` 파일이 잔류 → 재기동 시 `_recover_stale_processing()`으로 복원
- failcount++ 후 inbox/로 복원 (3회 미만)

**수동 대응:** 없음 (자동)

## 카테고리 C: 메시지 처리 장애

### C-1. inbox 파일 손상

**원인:** 원자적 쓰기 실패 (`.tmp` → `rename` 사이 장애), 디스크 오류

## 카테고리 D: 채널 발송 장애

### D-1. Telegram 발송 실패 (일시적)

**원인:** 네트워크 불안정, Telegram API 일시 장애

**영향:** 해당 메시지 재시도

- `_send_to_targets()`에서 실패한 target만 `failed_targets`로 분리
- retry_count++ 후 outbox/로 복원 → 재시도
- 3회(`MAX_OUTBOX_RETRIES`) 초과 시 error/ 격리

**수동 대응:**
```bash
mv messages/error/파일.json messages/outbox/
```

**코드 참조:**
- `outbox_sender.py:89-116` — 부분 실패 재시도 로직

### D-2. Telegram 발송 실패 (영구적)

**원인:** 봇 토큰 무효, chat_id 오류

**영향:** 모든 발송 실패

**자동 복구:**
- retry 3회 후 error/ 격리 (메시지별)

**수동 대응:**
```bash

**원인:** 동일 봇 토큰으로 여러 인스턴스 동시 실행

**영향:** polling 중단, 메시지 수신 불가

**자동 복구:** 없음

**수동 대응:**
```bash
# 다른 인스턴스 확인 및 종료

### D-4. Telegram HTML 파싱 에러

**원인:** 응답에 `<tag>` 형태 텍스트가 포함되어 HTML 파싱 실패

**영향:** 해당 메시지 발송 실패

**자동 복구:**
- `_send_to_targets()`에서 발송 실패 시 `parse_mode=None`으로 plain text 재시도

**수동 대응:** 없음 (자동 fallback)

**코드 참조:**
- `outbox_sender.py:95-97` — parse_mode 제거 후 재시도

### D-5. Slack 세션 반복 끊김

**원인:** Socket Mode WebSocket 연결 불안정

**영향:** 메시지 수신/발송 불안정

**자동 복구:**
- slack_bolt SDK 내장 자동 재연결

**수동 대응:**
```bash

```

### D-6. 부분 채널 실패

**원인:** 멀티채널 환경에서 한 채널만 발송 실패

**영향:** 실패 채널에만 미발송

**자동 복구:**
- 성공한 target은 제외, `failed_targets`만 retry_count++ 후 재시도

**수동 대응:** 없음 (자동)

**코드 참조:**

## 장애 시 알림 계획

모든 장애 알림은 `_send_alert()`를 통해 outbox/에 JSON 파일로 생성되며, 활성화된 모든 채널(Telegram, Slack)에 동시 브로드캐스트됩니다.

### 알림 발생 매트릭스

| 알림 | 트리거 조건 | 메시지 형식 | 관련 시나리오 | 코드 위치 |
|------|-----------|------------|-------------|----------|
| 작업 시작 | inbox → .processing 전환 시 | `[채널][사용자][키워드] 작업중입니다. (대기 N건)` | 정상 동작 | inbox_processor.py:181-187 |
| 처리 중 경과 | `progress_interval`초 간격 (기본 30초) | `[시스템] {label} 처리 중... ({시간} 경과)` | 정상 동작 | inbox_processor.py:438-453 |
| 세션 리셋 | `max_session_turns` 도달 시 | `[시스템] 세션이 자동 리셋되었습니다. (N건 처리 완료)` | 정상 동작 | inbox_processor.py:269-271 |
| 파일 격리 (처리) | 파일별 실패 3회 초과 | `[시스템] 메시지 처리 실패 (3회 연속). error/로 격리: {파일명}` | B-1, B-2, C-1 | inbox_processor.py:581-586 |

### 알림이 발생하지 않는 장애

다음 장애는 알림 메커니즘 자체가 동작할 수 없어 **채널 알림이 불가능**합니다. 로그 모니터링 또는 프로세스 감시가 필요합니다.

| 시나리오 | 알림 불가 이유 | 대체 감지 방법 |
|---------|--------------|--------------|
| A-1. 프로세스 비정상 종료 | 프로세스 자체가 종료됨 | PID 파일 + `kill -0` 체크, 외부 모니터링 |
| A-2. 프로세스 좀비화 | 폴링 루프가 멈춤 | `/status` 응답 없음으로 확인, 로그 갱신 중단 확인 |
| D-2. 채널 토큰 무효 | 발송 자체가 실패함 | `logs/gsd-orchestrator.log`에서 에러 확인 |
| D-3. Telegram Conflict | polling이 끊겨 수신 불가 | 로그에서 Conflict 에러 확인 |
| E-2. 파일 권한 오류 | outbox/ 쓰기 실패로 알림 파일 생성 불가 | 로그 확인 |
| E-3. 네트워크 단절 | 채널 API 호출 불가 | 로그 확인, 네트워크 모니터링 |
| E-4. 시스템 재부팅 | 프로세스 종료됨 | 외부 모니터링 (launchd/systemd) |

### 알림 흐름도

```
장애 발생
    │
    ├── 프로세스 생존? ──── No ──→ 알림 불가 (로그/외부 모니터링)
    │       │
    │      Yes
    │       │
    ├── 채널 연결 정상? ── No ──→ 알림 불가 (로그에 에러 기록)
    │       │
    │      Yes
    │       │
    ├── _send_alert() ──→ outbox/{ts}_system-alert.json 생성
    │       │
    │       └── outbox_sender (3초 폴링) ──→ 전채널 브로드캐스트
    │
    └── 알림 도달 확인: 채널에서 메시지 수신
```

### 에스컬레이션 단계

장애가 지속되면 알림이 다음 순서로 단계적으로 강화됩니다.

```
1단계: 파일별 재시도 (자동, 알림 없음)
    │   failcount 1~2회 — 파일을 inbox/로 복원하여 재처리
    │
    ▼
2단계: 파일 격리 알림 (자동)
    │   failcount 3회 — error/로 격리 + 채널 알림
    │   "[시스템] 메시지 처리 실패 (3회 연속). error/로 격리: {파일명}"
    │
    ▼
3단계: 쿨다운 진입 알림 (자동)
    │   전역 failcount 5회 — 쿨다운 모드 + 채널 알림
    │   "[시스템] 연속 실패 감지. N분 후 자동 재시도. 즉시 재개: /resume"
    │
    ▼
4단계: 쿨다운 장기 지속 경고 (자동, 1회)
    │   쿨다운 5시간 이상 — 추가 경고 (중복 방지: .cooldown-alerted)
    │   "[시스템] 쿨다운 5시간 이상 지속. 토큰 상태 확인 필요."
    │
    ▼
5단계: 수동 개입 필요
        /resume로 즉시 재개 또는 근본 원인 해결
```

### 외부 모니터링 연동 (권장)

프로세스 장애 등 채널 알림이 불가능한 상황을 감지하려면 외부 모니터링을 설정합니다.

**프로세스 생존 확인:**
```bash
# crontab에 등록 — 5분마다 프로세스 확인, 죽었으면 재시작
INSTANCE_ID=$(echo -n "/path/to/GSD-Orchestrator" | md5 -q | cut -c1-8)
PID_FILE="/tmp/gsd-orchestrator-${INSTANCE_ID}.pid"

*/5 * * * * [ -f "$PID_FILE" ] && kill -0 $(cat "$PID_FILE") 2>/dev/null || cd /path/to/GSD-Orchestrator && ./start.sh
```

**로그 기반 감지:**
```bash
# 최근 10분간 로그 갱신이 없으면 경고
find logs/gsd-orchestrator.log -mmin +10 -exec echo "로그 갱신 중단 — 프로세스 확인 필요" \;
```
