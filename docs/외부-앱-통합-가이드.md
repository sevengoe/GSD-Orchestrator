# 외부 앱 통합 가이드 (App Bridge)

GSD Orchestrator v0.7 부터 사용자 → 외부 비즈니스 앱 방향의 명령 라우팅을 지원한다. GSD 는 명령을 어떤 앱으로 보낼지 결정만 하고, 비즈니스 로직은 외부 앱이 처리한다.

> **이 문서**: 통합 방법을 학습하는 튜토리얼/예시 중심 가이드.
> **공식 사양 (계약)**: [`App-Bridge-연동-정의서.md`](./App-Bridge-연동-정의서.md) — 다른 언어/팀이 구현 가능하도록 필드/상태/에러를 명세한 RFC 형식 문서. **구현 시 정의서를 우선 참조**.
>
> GSD 자체에 비즈니스 명령을 추가하지 않는 이유는 [`docs/아키텍처.md`](./아키텍처.md) 의 App Bridge 섹션 참고.

## 개요

```
사용자 (텔레그램/슬랙)
  ↓ /sell d2 005490 100
GSD Orchestrator
  ├─ 슬래시 명령 매칭 → app_bridge.apps[*].command_prefix
  ├─ whitelist_user_ids 검증
  ├─ ack 메시지 즉시 발송
  └─ 외부 앱 디스패치
        ↓
       (file 모드) external_inbox/{app}/*.json 작성 → 외부 앱이 폴링
       (api 모드)  등록된 Python 핸들러 직접 호출
        ↓
       외부 앱 응답 → GSD outbox/*.json (command_id 포함)
        ↓
       GSD OutboxSender → 사용자 채널
```

GSD 는 다음 책임만 진다:
- 명령 prefix 매칭, whitelist 검증
- command_id 생성 및 correlation 추적
- 응답 timeout 알림 (60초 기본)
- 사용자 채널 발송

GSD 가 **하지 않는 것**:
- 명령의 의미 해석
- 외부 앱 healthcheck
- 외부 앱 직접 제어 (DB insert, 프로세스 호출 등)

## 통합 모드 선택

| 모드 | 적합한 경우 | 통신 방식 |
|---|---|---|
| **file** | 외부 앱이 별도 프로세스 (다른 언어 OK) | 디렉토리 폴링 |
| **api**  | 외부 앱이 GSD 와 같은 Python 프로세스 (라이브러리로 임베드) | `AppBridge.register()` |

두 모드는 동시 사용 가능하다. 앱 A는 file, 앱 B는 api 로 등록할 수 있다.

## file 모드 (독립 프로세스)

### config.yaml 등록

```yaml
app_bridge:
  enabled: true
  external_inbox_base: messages/external_inbox
  response_timeout_sec: 60
  apps:
    - name: mmm
      mode: file
      inbox_dir: messages/external_inbox/mmm
      command_prefix: ["/sell", "/reconcile", "/sell_panic"]
      whitelist_user_ids: ["12345678"]
      ack_message: "[MMM] 명령 접수 (ID={id})"
```

### 외부 앱 폴링 패턴 (Python 예시)

```python
import json, time, uuid
from datetime import datetime, timezone, timedelta
from pathlib import Path

KST = timezone(timedelta(hours=9))
INBOX = Path("/path/to/GSD-Orchestrator/messages/external_inbox/mmm")
OUTBOX = Path("/path/to/GSD-Orchestrator/messages/outbox")

def handle_command(cmd: dict) -> str:
    args = cmd["command"]["args"]
    prefix = cmd["command"]["prefix"]
    if prefix == "/sell":
        return f"매도 요청 등록됨: {' '.join(args)}"
    return f"미지원 명령: {prefix}"

def write_response(cmd: dict, response_text: str) -> None:
    now = datetime.now(KST)
    fname = f"{now.strftime('%Y%m%d_%H%M%S')}_{uuid.uuid4().hex[:8]}_mmm.json"
    src = cmd["source"]
    data = {
        "id": str(uuid.uuid4()),
        "command_id": cmd["command_id"],
        "source": src,
        "targets": [{
            "channel_type": src["channel_type"],
            "channel_id": src["channel_id"],
            "is_origin": True,
        }],
        "retry_count": 0,
        "keyword": "mmm",
        "request": None,
        "response": {
            "text": response_text,
            "parse_mode": None,
            "timestamp": now.isoformat(),
        },
    }
    tmp = OUTBOX / f".{fname}.tmp"
    tmp.write_text(json.dumps(data, ensure_ascii=False, indent=2))
    tmp.rename(OUTBOX / fname)

while True:
    INBOX.mkdir(parents=True, exist_ok=True)
    for f in sorted(INBOX.glob("*.json")):
        try:
            cmd = json.loads(f.read_text())
            text = handle_command(cmd)
            write_response(cmd, text)
        finally:
            f.unlink(missing_ok=True)
    time.sleep(0.5)
```

핵심 규칙:
- **원자적 쓰기 필수**: `.tmp` 작성 후 `rename` (GSD 의 CLAUDE.md 룰 준수)
- **command_id 응답에 포함**: GSD correlator 가 pending 에서 정상 제거하기 위함
- **targets 의 is_origin=True 설정**: origin 채널로만 전체 응답 발송

## api 모드 (in-process)

호스트 앱이 GSD 를 라이브러리로 임베드한 경우, 또는 같은 프로세스에서 비즈니스 핸들러를 등록할 때 사용한다.

### 등록 코드

```python
from gsd_orchestrator.config import Config
from gsd_orchestrator.orchestrator import Orchestrator

config = Config.load("config.yaml")
orch = Orchestrator(config)

async def mmm_handler(payload: dict) -> str:
    args = payload["command"]["args"]
    prefix = payload["command"]["prefix"]
    if prefix == "/portfolio":
        # 비즈니스 로직 실행
        return await fetch_portfolio_summary()
    return "미지원 명령"

orch.app_bridge.register("mmm_api", mmm_handler)
orch.run()  # 또는 await orch.start()
```

### config.yaml

```yaml
app_bridge:
  enabled: true
  apps:
    - name: mmm_api
      mode: api
      command_prefix: ["/portfolio", "/risk_report"]
      whitelist_user_ids: ["12345678"]
```

핸들러는 sync 또는 async 모두 가능. 반환 문자열이 자동으로 outbox 로 enqueue 되어 사용자에게 발송된다. 예외 발생 시 에러 메시지가 사용자에게 전달된다.

## config.yaml 등록 — 전체 옵션

```yaml
app_bridge:
  enabled: true
  external_inbox_base: messages/external_inbox  # file 모드 기본 경로
  response_timeout_sec: 60                       # 외부 앱 응답 대기 (초)
  ack_timeout_sec: 5                             # 사용자 ack 발송 시간
  max_args_length: 1024                          # args 결합 길이 상한
  apps:
    - name: foo                                  # 필수: 앱 식별자
      mode: file                                  # file | api
      inbox_dir: messages/external_inbox/foo     # file 모드만 (생략 시 기본값)
      command_prefix: ["/foo", "/bar"]            # 매칭할 슬래시 prefix 목록
      whitelist_user_ids: ["12345"]              # 빈 리스트 = 전체 허용
      ack_message: "[foo] 접수 (ID={id})"         # 사용자에게 즉시 발송할 ack
```

### 시작 시 검증

- `command_prefix` 가 다른 앱과 충돌하면 `ValueError` 로 시작 실패
- prefix 가 `/` 로 시작하지 않으면 `ValueError`
- `mode` 가 `file` 또는 `api` 가 아니면 `ValueError`

## JSON 스키마

### external_inbox/{app}/*.json (GSD → 외부 앱)

```json
{
  "id": "msg-uuid",
  "command_id": "correlation-uuid",
  "timestamp": "2026-05-04T08:30:00+09:00",
  "source": {
    "channel_type": "telegram",
    "channel_id": "123456",
    "user_id": "12345678",
    "user_name": "kbj"
  },
  "command": {
    "raw": "/sell d2 005490 100",
    "prefix": "/sell",
    "args": ["d2", "005490", "100"]
  },
  "target_app": "mmm",
  "reply_to_outbox": "/abs/path/to/messages/outbox"
}
```

### outbox/*.json (외부 앱 → GSD → 사용자)

```json
{
  "id": "reply-uuid",
  "command_id": "correlation-uuid",
  "source": { /* 원본 source 그대로 */ },
  "targets": [
    {"channel_type": "telegram", "channel_id": "123456", "is_origin": true}
  ],
  "retry_count": 0,
  "keyword": "mmm_reply",
  "request": null,
  "response": {
    "text": "매도 요청 등록됨, ID=42",
    "parse_mode": null,
    "timestamp": "2026-05-04T08:30:01+09:00"
  }
}
```

`command_id` 는 GSD correlator 가 pending 추적에 사용. 응답에 반드시 포함하라.

## 에러 / 타임아웃 / 권한

| 상황 | GSD 동작 |
|---|---|
| `command_prefix` 매칭 실패 | App Bridge 미적용 — 일반 GSD 흐름 (Claude headless) |
| `whitelist_user_ids` 거부 | "권한 없음" 메시지 발송, 외부 앱 호출 안 함 |
| args 길이 초과 | "입력이 너무 깁니다" 거부 |
| args 위험 문자 (`;`, `\|`, `` ` ``, `$()`) 포함 | "허용되지 않는 문자" 거부 |
| `response_timeout_sec` 초과 | "외부 앱 응답 없음" 알림 발송 (best-effort: 늦게 도착해도 `[지연 응답]` prefix 로 전달) |
| 외부 앱이 잘못된 JSON 작성 | OutboxSender 가 `messages/error/` 로 격리 (기존 동작) |

## 예제: echo 앱

`tests/fixtures/echo_app.py` 가 file 모드 reference 구현이다. 받은 args 를 그대로 응답한다.

```bash
# 1) config.yaml 에 등록
cat >> config.yaml <<'EOF'
app_bridge:
  enabled: true
  apps:
    - name: echo
      mode: file
      command_prefix: ["/echo"]
      whitelist_user_ids: []
      ack_message: "[echo] ack id={id}"
EOF

# 2) GSD 시작
./start.sh

# 3) echo_app 폴링 시작 (별도 터미널)
.venv/bin/python -c "
import sys; sys.path.insert(0,'src')
from tests.fixtures.echo_app import run_loop
from pathlib import Path
run_loop(
    inbox_dir=Path('messages/external_inbox/echo'),
    outbox_dir=Path('messages/outbox'),
    stop_after=600,
)"

# 4) 텔레그램에서 /echo hello 전송 → "hello" 응답 도달
```

## 채널별 동작 차이

| 채널 | App Bridge 동작 메커니즘 | 추가 코드 |
|---|---|---|
| Telegram | `filters.COMMAND` 이 슬래시 명령을 일반 메시지에서 분리 → 미등록 명령용 fallback 핸들러 명시 | `channels/telegram.py:85-90` |
| Slack | Socket Mode `event("message")` 가 슬래시 텍스트를 일반 메시지로 그대로 전달 → 자동 라우팅 | 없음 (변경 0줄) |

> ⚠️ **Slack workspace 가 같은 prefix 를 native slash command 로 등록하면**, Slack 이 사용자 입력을 별도 HTTPS webhook 으로 보내고 GSD 의 Socket Mode 이벤트로는 도달하지 않는다 → App Bridge 라우팅 안 됨. Slack App 설정의 "Slash Commands" 와 prefix 가 겹치지 않도록 주의. 자세한 내용은 [정의서 §12.A](./App-Bridge-연동-정의서.md#12a-채널별-슬래시-명령-도달-메커니즘) 참고.
