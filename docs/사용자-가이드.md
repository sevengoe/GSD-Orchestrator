# GSD Orchestrator 사용자 가이드

## 사전 요구사항

### 1. Claude Code (필수)

GSD Orchestrator는 **Claude Code CLI**를 통해 AI 작업을 수행합니다.

| 플랜 | 지원 여부 | 비고 |
|------|-----------|------|
| **Claude Pro** | 지원 | 일일 사용량 제한 있음 |
| **Claude Max** | 지원 (권장) | 높은 사용량 한도, 장시간 자율 작업에 적합 |
| **Claude Free** | 미지원 | Claude Code CLI 사용 불가 |

> **핵심:** API 키가 아닌 Claude Code 구독(`claude` CLI)으로 동작합니다.

```bash
# macOS
brew install claude-code

# npm (모든 플랫폼)
npm install -g @anthropic-ai/claude-code

# 설치 후 로그인
claude
```

### 2. GSD Framework (GSD 트랙 사용 시)

단순 질문/답변만 사용한다면 설치 불필요.

```bash
npx get-shit-done-cc --claude --global

# 설치 확인
ls ~/.claude/commands/gsd/
```

### 3. Python 3.11+

```bash
python3 --version   # 3.11 이상
```

### 4. 채널 설정 (최소 하나)

**Telegram:**
1. [@BotFather](https://t.me/BotFather)에서 `/newbot`으로 봇 생성
2. Bot Token 복사
3. 봇에게 메시지 보낸 후 Chat ID 확인:
```bash
curl -s "https://api.telegram.org/bot<TOKEN>/getUpdates" | python3 -m json.tool
# result[0].message.chat.id
```

**Slack (optional, Socket Mode 방식):**

Socket Mode는 WebSocket 기반으로 공개 URL/서버 없이 로컬 PC에서 메시지를 수신할 수 있습니다.

1. [Slack API](https://api.slack.com/apps)에서 앱 생성
2. **Socket Mode** 활성화 → App-Level Token(`xapp-...`) 발급
3. Bot Token Scopes 추가: `channels:history`, `channels:join`, `channels:read`, `chat:write`, `users:read`
4. **Event Subscriptions** 활성화 → `message.channels` 이벤트 구독
5. 앱 설치(Install to Workspace) → Bot Token(`xoxb-...`) 확인
6. 채널에 봇 추가: 채널 상단 이름 클릭 → 통합(Integrations) → 앱 추가(Add apps) → 봇 선택
7. 채널 ID 확인: 채널 이름 클릭 → 하단에 표시되는 ID(`C...`) 복사 → `.env`에 입력

## 설치

```bash
git clone git@github.com:sevengoe/GSD-Orchestrator.git
cd GSD-Orchestrator

# 초기 설정 (venv 생성, 의존성 설치, 디렉토리 생성)
./setup.sh                # 기본 (Telegram만)
./setup.sh --slack        # Slack 포함
```

`.env` 편집 (setup.sh가 .env.example에서 자동 생성):

```env
TELEGRAM_BOT_TOKEN=1234567890:AABBccDD...
TELEGRAM_CHAT_ID=123456789

# Slack 사용 시 (optional)
SLACK_BOT_TOKEN=xoxb-your-token
SLACK_APP_TOKEN=xapp-your-token
SLACK_CHANNEL_ID=C12345ABCDE
```

Slack 사용 시 `config.yaml`에서 채널 활성화:

```yaml
channels:
  slack:
    enabled: true       # false → true
```

## 사용법

### 명령어

| 명령어 | 동작 |
|--------|------|
| 일반 메시지 | 자동 분류 후 최적 트랙으로 라우팅 |
| `/gsd <작업>` | GSD 워크플로우로 강제 라우팅 |
| `/status` | 쿨다운 상태, 토큰 사용량 |
| `/reset` | 다음 요청부터 새 세션 |
| `/resume` | 쿨다운 즉시 해제 |

### 일반 메시지 (Simple Track)

```
"이 함수 시간복잡도 분석해줘"
"Docker compose 파일 만들어줘"
```

### GSD 작업 (GSD Track)

```
/gsd 결제 모듈 Stripe 연동 구현해줘
/gsd 테스트 커버리지 80% 이상으로 올려줘
```

### GSD 블로킹 (Ask Human)

GSD 작업 중 판단이 필요하면 질문이 옵니다. 답변을 보내면 자동 재개:

```
GSD 블로킹: Stripe vs Toss 중 어떤 결제 게이트웨이를 사용할까요?
→ "Stripe로 해줘"
```

### 멀티채널 동작

Telegram과 Slack 모두 활성화 시:

- **수신 확인**: 모든 채널에 `[Telegram][김철수][결제모듈] 작업중입니다.`
- **처리 완료**: origin 채널에 전체 응답, 상대 채널에 500자 요약본
- **시스템 알림**: 모든 채널에 동일

## 설정

### config.yaml

```yaml
channels:
  telegram:
    enabled: true
    chat_id: ""                    # 빈 문자열이면 .env 사용
  slack:
    enabled: false
    channel_id: ""

broadcast:
  snippet_length: 500              # 상대 채널 요약 길이 (0이면 전체)

polling:
  inbox_check_interval: 10
  outbox_interval: 3
  progress_interval: 30            # 0이면 비활성화

archive:
  message_retention_days: 30

claude:
  timeout: 300
  cooldown_retry_minutes: 10
  max_session_turns: 20
  working_dir: workspace

gsd:
  enabled: true
  auto_classify: true
  classify_model: haiku
```

### 주요 시나리오

| 목적 | 설정 |
|------|------|
| 중간 보고 끄기 | `progress_interval: 0` |
| GSD 없이 챗봇만 | `gsd.enabled: false`, `gsd.auto_classify: false` |
| 세션 길게 유지 | `claude.max_session_turns: 50` |
| 브로드캐스트 전체 발송 | `broadcast.snippet_length: 0` |

## 메시지 디렉토리

모든 메시지는 JSON 파일로 관리됩니다. DB 없이 파일 이동 = 상태 전이.

```
messages/
├── inbox/       ← 수신된 요청 (처리 대기)
├── outbox/      ← 처리 완료 (발송 대기)
├── sent/        ← 발송 완료
├── error/       ← 처리/발송 실패 (격리)
└── archive/     ← 일자별 보관 (자동 만료 삭제)
    └── YYYY-MM-DD/
```

| 디렉토리 | 역할 | 파일 상태 |
|---|---|---|
| `inbox/` | 사용자 요청 대기열 | `.json` → `.json.processing` (처리 중) |
| `outbox/` | 채널 발송 대기열 | `.json` → `.json.sending` (발송 중) |
| `sent/` | 발송 완료 보관 | 매시간 archive/로 이동 |
| `error/` | 실패 메시지 격리 | 수동 복구: `mv error/파일 inbox/` |
| `archive/` | 일자별 보관 | `message_retention_days` 초과 시 자동 삭제 |

### 메시지 흐름

```
inbox/ → .processing → outbox/ → .sending → sent/ → archive/
                ↓                     ↓
           3회 실패 → error/     3회 실패 → error/
```

### 수동 조작

```bash
# 격리된 메시지 재처리
mv messages/error/파일.json messages/inbox/

# 발송 실패 메시지 재발송
mv messages/error/파일.json messages/outbox/

# 현재 대기 상태 확인
ls messages/inbox/     # 처리 대기
ls messages/outbox/    # 발송 대기
ls messages/error/     # 격리된 메시지
```

## 트러블슈팅

### "Conflict: terminated by other getUpdates request"

**인스턴스 1개 = 봇 토큰 1개** — 동일 봇 토큰으로 여러 인스턴스를 동시에 실행할 수 없습니다.

- **Telegram**: 같은 봇 토큰으로 두 프로세스가 polling하면 Conflict 에러 발생 (기기/경로 무관)
- **Slack**: 같은 앱 토큰으로 여러 인스턴스 연결 시 메시지가 랜덤 분산되어 정상 작동 불가

여러 인스턴스를 운영하려면 각각 별도의 봇/앱을 생성하세요.

단일 인스턴스 확인:
```bash
./stop.sh && ps aux | grep gsd_orchestrator && ./start.sh
```

### "claude: command not found"

```bash
npm install -g @anthropic-ai/claude-code
```

## 외부 앱 연동

외부 애플리케이션과 두 가지 모드로 연동할 수 있습니다.

| 모드 | 방식 | 기동 |
|------|------|------|
| **Library** | Python API (`send()`, `on_result()`) | 호스트 앱 코드에서 `run()` 또는 `start()` 호출 |
| **독립 프로세스** | 파일 기반 (`inbox/`, `sent/`) | `./start.sh`로 별도 프로세스 기동 |

두 모드 모두 GSD-Orchestrator의 기동이 필요합니다. Library 모드는 호스트 앱 프로세스 안에서 함께 실행되고, 독립 프로세스 모드는 별도 프로세스로 먼저 기동해야 합니다.

**Library 모드:**
```bash
pip install git+https://github.com/sevengoe/GSD-Orchestrator.git
```
- **`ChannelSender`** — 호스트 앱 → 채널 발송 (`send()`, `enqueue()`)
- **`on_result(callback)`** — 처리 결과 콜백 (success/error/blocked)

**독립 프로세스 모드:**
- 요청: `messages/inbox/`에 JSON 파일 작성 (원자적 쓰기)
- 결과: `messages/sent/`에서 처리 완료 파일 폴링
- 별도 패키지 설치 불필요

상세 사용법은 [`docs/외부-연동-가이드.md`](외부-연동-가이드.md)를 참조하세요.
