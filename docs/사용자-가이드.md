# GSD Orchestrator 사용자 가이드

## 사전 요구사항

### 1. Claude Code (필수)

GSD Orchestrator는 **Claude Code CLI**를 통해 AI 작업을 수행합니다.

| 플랜 | 지원 여부 | 비고 |
|------|-----------|------|
| **Claude Pro** | 지원 | 일일 사용량 제한 있음 |
| **Claude Max** | 지원 (권장) | 높은 사용량 한도, 장시간 자율 작업에 적합 |
| **Claude Free** | 미지원 | Claude Code CLI 사용 불가 |

> **핵심:** API 키가 아닌 Claude Code 구독(`claude` CLI)으로 동작합니다.

```bash
# macOS
brew install claude-code

# npm (모든 플랫폼)
npm install -g @anthropic-ai/claude-code

# 설치 후 로그인
claude
```

### 2. GSD Framework (GSD 트랙 사용 시)

단순 질문/답변만 사용한다면 설치 불필요.

[GSD(Getting Shit Done)](https://github.com/cyanheads/get-shit-done-cc)는 Claude Code 위에서 동작하는 자율형 개발 프레임워크입니다. 설치하면 `~/.claude/commands/gsd/` 에 슬래시 명령어(`/gsd:do`, `/gsd:next` 등)가 추가되며, GSD Orchestrator의 `/gsd` 명령어가 이 명령어들을 자동으로 호출하여 기획 → 작업 분할 → 구현 → 검증 워크플로우를 수행합니다.

```bash
npx get-shit-done-cc --claude --global

# 설치 확인
ls ~/.claude/commands/gsd/
```

### 3. Python 3.11+

```bash
python3 --version   # 3.11 이상
```

### 4. 채널 설정 (최소 하나)

**Telegram:**
1. [@BotFather](https://t.me/BotFather)에서 `/newbot`으로 봇 생성
2. Bot Token 복사
3. 봇에게 메시지 보낸 후 Chat ID 확인:
```bash
curl -s "https://api.telegram.org/bot<TOKEN>/getUpdates" | python3 -m json.tool
# result[0].message.chat.id
```

**Slack (optional, Socket Mode 방식):**

Socket Mode는 WebSocket 기반으로 공개 URL/서버 없이 로컬 PC에서 메시지를 수신할 수 있습니다.

1. [Slack API](https://api.slack.com/apps)에서 앱 생성
2. **Socket Mode** 활성화 → App-Level Token(`xapp-...`) 발급
3. Bot Token Scopes 추가: `channels:history`, `channels:join`, `channels:read`, `chat:write`, `files:read`, `users:read`
4. **Event Subscriptions** 활성화 → `message.channels`, `file_shared` 이벤트 구독
5. 앱 설치(Install to Workspace) → Bot Token(`xoxb-...`) 확인
6. 채널에 봇 추가: 채널 상단 이름 클릭 → 통합(Integrations) → 앱 추가(Add apps) → 봇 선택
7. 채널 ID 확인: 채널 이름 클릭 → 하단에 표시되는 ID(`C...`) 복사 → `.env`에 입력

## 설치

```bash
git clone git@github.com:sevengoe/GSD-Orchestrator.git
cd GSD-Orchestrator

# 초기 설정 (venv 생성, 의존성 설치, 디렉토리 생성)
./setup.sh                # config.yaml의 slack.enabled가 true이면 Slack 자동 설치
./setup.sh --slack        # 명시적으로 Slack 포함
```

`.env` 편집 (setup.sh가 .env.example에서 자동 생성):

```env
TELEGRAM_BOT_TOKEN=1234567890:AABBccDD...
TELEGRAM_CHAT_ID=123456789

# Slack 사용 시 (optional)
SLACK_BOT_TOKEN=xoxb-your-token
SLACK_APP_TOKEN=xapp-your-token
SLACK_CHANNEL_ID=C12345ABCDE
```

Slack 사용 시 `config.yaml`에서 채널 활성화:

```yaml
channels:
  slack:
    enabled: true       # false → true
```

## 사용법

### 명령어

| 명령어 | 동작 |
|--------|------|
| 일반 메시지 | 자동 분류 후 최적 트랙으로 라우팅 |
| `/gsd <작업>` | GSD 워크플로우로 강제 라우팅 |
| `/status` | 쿨다운 상태, 토큰 사용량 |
| `/reset` | 다음 요청부터 새 세션 |
| `/resume` | 쿨다운 즉시 해제 |

### 일반 메시지 (Simple Track)

```
"이 함수 시간복잡도 분석해줘"
"Docker compose 파일 만들어줘"
```

### GSD 작업 (GSD Track)

```
/gsd 결제 모듈 Stripe 연동 구현해줘
/gsd 테스트 커버리지 80% 이상으로 올려줘
```

### GSD 블로킹 (Ask Human)

GSD 작업 중 판단이 필요하면 질문이 옵니다. 답변을 보내면 자동 재개:

```
GSD 블로킹: Stripe vs Toss 중 어떤 결제 게이트웨이를 사용할까요?
→ "Stripe로 해줘"
```

### 첨부파일

텔레그램/슬랙에서 파일을 첨부하면 자동으로 텍스트를 추출하여 Claude에 전달합니다.

**지원 파일**: txt, md, pdf (최대 1MB)

```
사용자: [report.pdf 첨부]
봇: 아래 내용이 맞는지 확인해주세요:
    [첨부파일: report.pdf]
    (정제된 텍스트 내용...)
사용자: 네
봇: (Claude가 텍스트를 분석하여 응답)
```

- 지원하지 않는 파일(png, xlsx 등) → "txt, md, pdf 파일만 지원합니다" 안내
- PDF 이미지 전용 → "이미지로 구성되어 텍스트 추출이 불가합니다" 안내
- 암호화 PDF → "이 PDF는 보호되어 열 수 없습니다" 안내
- 파일 자체는 메타데이터만 보관, 임시 파일은 즉시 삭제

설정 (`config.yaml`):

```yaml
attachments:
  allowed_extensions: [txt, md, pdf]   # 허용 확장자
  max_file_size: 1048576               # 최대 파일 크기 (바이트, 1MB)
  temp_dir: messages/attachments       # 임시 저장 경로
  reject_message: "txt, md, pdf 파일만 지원합니다."
```

### 멀티채널 동작

Telegram과 Slack 모두 활성화 시:

- **수신 확인**: 모든 채널에 `[Telegram][김철수][결제모듈] 작업중입니다.`
- **처리 완료**: origin 채널에 전체 응답, 상대 채널에 500자 요약본
- **시스템 알림**: 모든 채널에 동일

## 설정

### config.yaml

```yaml
channels:
  telegram:
    enabled: true
    chat_id: ""                    # 빈 문자열이면 .env 사용
  slack:
    enabled: false
    channel_id: ""

broadcast:
  snippet_length: 500              # 상대 채널 요약 길이 (0이면 전체)

polling:
  inbox_check_interval: 10
  outbox_interval: 3
  progress_interval: 30            # 0이면 비활성화

archive:
  message_retention_days: 30

claude:
  timeout: 300
  cooldown_retry_minutes: 10
  max_session_turns: 20
  working_dir: workspace

gsd:
  enabled: true
  auto_classify: true
  classify_model: haiku

alert:
  quiet_start: -1                  # 알림 무음 시작 시각 (0~23, -1이면 비활성)
  quiet_end: -1                    # 알림 무음 종료 시각 (예: 23~6 → 23시~06시 무음)

mcp:
  enabled: true                    # MCP 파일시스템 서버 활성화
  allowed_directories: []          # 허용 디렉토리 (비어있으면 claude.working_dir 사용)

attachments:
  allowed_extensions: [txt, md, pdf]   # 허용 확장자
  max_file_size: 1048576               # 최대 파일 크기 (바이트, 1MB)
  temp_dir: messages/attachments       # 임시 저장 경로
  reject_message: "txt, md, pdf 파일만 지원합니다."
```

### MCP 파일시스템 서버

MCP(Model Context Protocol) 서버를 활성화하면 Claude Code가 `--dangerously-skip-permissions` 대신 **허용된 디렉토리 내에서만** 표준화된 도구로 파일 작업을 수행합니다.

```yaml
mcp:
  enabled: true                    # MCP 서버 활성화 (기본값: false)
  allowed_directories:             # 허용 디렉토리 목록
    - workspace                    # 상대 경로는 프로젝트 루트 기준
    - /home/user/data              # 절대 경로도 가능
```

- `allowed_directories`가 비어있으면 `claude.working_dir` (기본값: `workspace`)이 자동 사용됩니다
- `mcp-config.json`이 프로젝트 루트에 자동 생성되며, Claude CLI가 이를 참조합니다
- 심링크/`../` 경로 우회 공격이 차단됩니다

**제공 도구 (12개)**:

| 도구 | 설명 |
|------|------|
| `read_file` | 파일 읽기 (head/tail 지원) |
| `write_file` | 원자적 파일 쓰기 |
| `list_directory` | 디렉토리 목록 |
| `create_directory` | 디렉토리 생성 |
| `get_file_info` | 파일 메타데이터 |
| `move_file` | 파일 이동 |
| `delete_file` | 파일 삭제 |
| `search_files` | glob 패턴 검색 |
| `grep_files` | 정규식 내용 검색 |
| `directory_tree` | 재귀 트리 구조 |
| `read_multiple_files` | 다중 파일 읽기 |
| `list_allowed_directories` | 허용 경로 목록 |

### 주요 시나리오

| 목적 | 설정 |
|------|------|
| 중간 보고 끄기 | `progress_interval: 0` |
| GSD 없이 챗봇만 | `gsd.enabled: false`, `gsd.auto_classify: false` |
| 세션 길게 유지 | `claude.max_session_turns: 50` |
| 브로드캐스트 전체 발송 | `broadcast.snippet_length: 0` |
| 야간 알림 무음 (23시~06시) | `alert.quiet_start: 23`, `alert.quiet_end: 6` |
| MCP 파일 접근 제어 | `mcp.enabled: true`, `mcp.allowed_directories: [workspace]` |

## 메시지 디렉토리

모든 메시지는 JSON 파일로 관리됩니다. DB 없이 파일 이동 = 상태 전이.

```
messages/
├── inbox/       ← 수신된 요청 (처리 대기)
├── outbox/      ← 처리 완료 (발송 대기)
├── sent/        ← 발송 완료
├── error/       ← 처리/발송 실패 (격리)
├── archive/     ← 일자별 보관 (자동 만료 삭제)
│   └── YYYY-MM-DD/
├── attachments/ ← 첨부파일 임시 저장 (처리 후 즉시 삭제)
├── workqueue/   ← GSD 작업 큐 (단위별 순차 실행)
└── plan/        ← GSD 계획서 (.gsd-plan.md)
```

| 디렉토리 | 역할 | 파일 상태 |
|---|---|---|
| `inbox/` | 사용자 요청 대기열 | `.json` → `.json.processing` (처리 중) |
| `outbox/` | 채널 발송 대기열 | `.json` → `.json.sending` (발송 중) |
| `sent/` | 발송 완료 보관 | 매시간 archive/로 이동 |
| `error/` | 실패 메시지 격리 | 수동 복구: `mv error/파일 inbox/` |
| `archive/` | 일자별 보관 | `message_retention_days` 초과 시 자동 삭제 |
| `workqueue/` | GSD 작업 큐 | 단위별 `.json`, 순차 실행 후 삭제 |
| `plan/` | GSD 계획서 | `.gsd-plan.md`, 전체 완료 시 삭제 |

### 메시지 흐름

```
inbox/ → .processing → outbox/ → .sending → sent/ → archive/
                ↓                     ↓
           3회 실패 → error/     3회 실패 → error/
```

### 수동 조작

```bash
# 격리된 메시지 재처리
mv messages/error/파일.json messages/inbox/

# 발송 실패 메시지 재발송
mv messages/error/파일.json messages/outbox/

# 현재 대기 상태 확인
ls messages/inbox/     # 처리 대기
ls messages/outbox/    # 발송 대기
ls messages/error/     # 격리된 메시지
```

## 트러블슈팅

### "Conflict: terminated by other getUpdates request"

**인스턴스 1개 = 봇 토큰 1개** — 동일 봇 토큰으로 여러 인스턴스를 동시에 실행할 수 없습니다.

- **Telegram**: 같은 봇 토큰으로 두 프로세스가 polling하면 Conflict 에러 발생 (기기/경로 무관)
- **Slack**: 같은 앱 토큰으로 여러 인스턴스 연결 시 메시지가 랜덤 분산되어 정상 작동 불가

여러 인스턴스를 운영하려면 각각 별도의 봇/앱을 생성하세요.

단일 인스턴스 확인:
```bash
./stop.sh && ps aux | grep gsd_orchestrator && ./start.sh
```

### Slack "channel_not_found"

`.env`의 `SLACK_CHANNEL_ID`가 실제 채널 ID와 다를 때 발생합니다. 채널 ID는 비슷하게 생긴 다른 ID와 혼동하기 쉬우므로 API로 확인하세요:

```bash
source .env
curl -s "https://slack.com/api/conversations.info?channel=$SLACK_CHANNEL_ID" \
 -H "Authorization: Bearer $SLACK_BOT_TOKEN" | python3 -m json.tool
```

- `"ok": true` → 채널 ID 정상
- `"error": "channel_not_found"` → Slack에서 채널명 클릭 → 팝업 하단의 ID(`C...`)를 `.env`에 다시 입력

### Python 버전 체크 실패 ("Python 3.11 이상이 필요합니다")

macOS 시스템 기본 `python3`가 3.9인 경우 발생합니다. Homebrew로 3.11 이상이 설치되어 있으면 `setup.sh`가 자동으로 `python3.13` → `python3.12` → `python3.11` 순서로 탐색합니다.

```bash
# Homebrew Python 설치
brew install python@3.13
```

### venv 생성 시 ensurepip 오류

Homebrew Python에서 `ensurepip`이 동작하지 않는 경우입니다. `setup.sh`가 자동으로 `--without-pip` + pip 수동 설치로 fallback합니다. 수동 설치 시:

```bash
/opt/homebrew/bin/python3.13 -m venv .venv --without-pip
curl -sS https://bootstrap.pypa.io/get-pip.py | .venv/bin/python3
```

### SSL 인증서 검증 실패 (프록시/VPN 환경)

```
SSL: CERTIFICATE_VERIFY_FAILED - self-signed certificate in certificate chain
```

회사 VPN, 방화벽 등 SSL 인터셉트 프록시 환경에서 발생합니다. Python의 `certifi`가 프록시의 자체 서명 인증서를 거부하기 때문입니다. `setup.sh`가 자동 감지하여 조치하지만, 수동 설치 시:

```bash
source .venv/bin/activate
pip install pip-system-certs
./restart.sh
```

### 메시지 수신 지연 / 응답 없음

PC가 **절전 모드(Sleep)** 에 들어가면 네트워크가 비활성화되면서 Telegram/Slack polling이 중단됩니다. 프로세스는 살아 있지만 메시지 수신이 멈추고, 절전에서 깨어난 후 밀린 메시지가 한꺼번에 처리됩니다.

**`./start.sh`는 자동으로 절전 방지를 설정합니다:**
- **macOS**: `caffeinate`로 프로세스 실행 중 절전 진입 차단
- **Linux**: `systemd-inhibit`로 idle 절전 차단

오케스트레이터를 `./stop.sh`로 중지하면 절전 방지도 함께 해제됩니다.

절전 방지가 동작하지 않는 경우 (수동 설정):
```bash
# macOS — 절전 비활성화
sudo pmset -a disablesleep 1
# 복구: sudo pmset -a disablesleep 0

# Linux — 절전 비활성화
sudo systemctl mask sleep.target suspend.target hibernate.target hybrid-sleep.target
```

> 수신 지연이 감지되면 시스템이 자동으로 지연 사유를 안내합니다.

## 외부 앱 연동

외부 애플리케이션과 두 가지 모드로 연동할 수 있습니다.

| 모드 | 방식 | 기동 |
|------|------|------|
| **Library** | Python API (`send()`, `on_result()`) | 호스트 앱 코드에서 `run()` 또는 `start()` 호출 |
| **독립 프로세스** | 파일 기반 (`inbox/`, `sent/`) | `./start.sh`로 별도 프로세스 기동 |

두 모드 모두 GSD-Orchestrator의 기동이 필요합니다. Library 모드는 호스트 앱 프로세스 안에서 함께 실행되고, 독립 프로세스 모드는 별도 프로세스로 먼저 기동해야 합니다.

**Library 모드:**
```bash
pip install git+https://github.com/sevengoe/GSD-Orchestrator.git
```
- **`ChannelSender`** — 호스트 앱 → 채널 발송 (`send()`, `enqueue()`)
- **`on_result(callback)`** — 처리 결과 콜백 (success/error/blocked)

**독립 프로세스 모드:**
- 요청: `messages/inbox/`에 JSON 파일 작성 (원자적 쓰기)
- 결과: `messages/sent/`에서 처리 완료 파일 폴링
- 별도 패키지 설치 불필요

상세 사용법은 [`docs/외부-연동-가이드.md`](외부-연동-가이드.md)를 참조하세요.

## 외부 비즈니스 앱 통합 (App Bridge, v0.7+)

사용자가 보낸 슬래시 명령(예: `/sell`, `/portfolio`)을 외부 비즈니스 앱으로 라우팅하는 인프라. GSD 본체는 명령의 의미를 모르고, 라우팅·권한·timeout 만 처리합니다.

config.yaml 등록 예시:

```yaml
app_bridge:
  enabled: true
  response_timeout_sec: 60
  apps:
    - name: mmm
      mode: file                 # file | api
      command_prefix: ["/sell", "/reconcile"]
      whitelist_user_ids: ["12345678"]
      ack_message: "[MMM] 명령 접수 (ID={id})"
```

- **file 모드**: 외부 앱이 `messages/external_inbox/{name}/` 폴링 → `messages/outbox/` 응답
- **api 모드**: 같은 Python 프로세스에서 `orchestrator.app_bridge.register(name, handler)` 등록

GSD 메타 명령(`/help`, `/gsd`, `/status`, `/reset`, `/resume`, `/retry`)은 예약어로 등록 불가.

- 상세 사용법: [`docs/외부-앱-통합-가이드.md`](외부-앱-통합-가이드.md) (튜토리얼/예시)
- 공식 계약 (다른 언어 구현용): [`docs/App-Bridge-연동-정의서.md`](App-Bridge-연동-정의서.md) (RFC 형식 사양, JSON Schema, conformance checklist)
