# 에이전틱 AI와 GSD Orchestrator

## 에이전틱 AI란?

에이전틱 AI(Agentic AI)는 사용자가 매 단계를 지시하지 않아도 **목표만 주어지면 스스로 계획하고, 도구를 사용하고, 오류를 수정하며, 최종 결과를 달성하는 AI 시스템**을 말합니다. 기존의 단답형 챗봇이 "질문 → 답변" 1회성 상호작용에 머무는 반면, 에이전틱 AI는 멀티스텝 루프를 자율적으로 수행합니다.

업계에서는 이를 **AI Agent** 또는 **Autonomous Agent**라 부르며, 2025년 이후 소프트웨어 개발 분야에서 "Engineering Agent"라는 이름으로 빠르게 확산되고 있습니다.

### 핵심 요소 5가지

| 요소 | 설명 |
|------|------|
| **자율성 (Autonomy)** | 최종 목표만 받으면 중간 단계를 스스로 설계하고 실행 |
| **비동기 장기 실행 (Long-running)** | 수 분~수 시간 단위 작업을 백그라운드에서 수행 |
| **상태 복원력 (Resilience)** | 오류·중단 시 진행 상태를 보존하고 자동 재개 |
| **도구 사용 (Tool Use)** | 파일 시스템, 터미널, API 등 외부 환경과 직접 상호작용 |
| **문맥 최적화 (Context Management)** | 비용·효율을 고려한 지능형 라우팅과 세션 격리 |

## GSD Orchestrator는 이 요소를 어떻게 구현하는가

### 1. 자율성 — 목표에서 코드까지

사용자가 메신저로 "결제 모듈 연동해 줘"라고 보내면, 시스템이 내부적으로 다음을 수행합니다:

```
사용자 메시지 → 자동 분류 → 요구 분석 → 작업 분할(WBS) → 단위별 코딩 → 검증 → 결과 보고
```

복잡한 작업은 자동으로 `.gsd-plan.md` 계획서를 작성하고, 승인 후 작업 큐(Work Queue)로 분할하여 순차 실행합니다. 사용자가 각 단계를 지시할 필요가 없습니다.

### 2. 비동기 장기 실행 — Fire & Forget

**동기식 챗봇**: 사용자가 응답 완료까지 화면을 주시해야 합니다.

**GSD Orchestrator**: 명령을 내리고 일상으로 복귀합니다. `inbox_processor`가 10초 간격으로 폴링하며 백그라운드에서 작업을 수행하고, 완료 시 메신저로 결과를 보고합니다.

```
사용자: 메시지 전송 → 즉시 "작업중입니다" 확인 → 일상 복귀
시스템: inbox 수신 → 분류 → Claude Code 실행 → 30초 간격 진행 알림 → 완료 보고
```

작업이 10분을 초과할 것으로 판단되면, 계획서를 먼저 제출받고 단위별로 분할 실행하므로 타임아웃 시에도 완료된 단위의 결과가 보존됩니다.

### 3. 상태 복원력 — 파일 기반 상태 머신

DB 없이 **파일 이동 = 상태 전이** 원칙으로 동작합니다:

```
inbox/msg.json                  ← 대기
inbox/msg.json.processing       ← 처리 중 (lock)
outbox/msg.json                 ← 응답 생성 완료
outbox/msg.json.sending         ← 발송 중 (lock)
sent/msg.json                   ← 발송 완료
```

이 설계 덕분에:
- **프로세스가 죽어도** `.processing` 파일이 남아 있으면 재시작 시 `inbox/`로 복원하여 재처리
- **네트워크 장애 시** `outbox/`의 `retry_count`를 증가시키며 최대 3회 재시도
- **반복 실패 시** `error/`로 격리하여 다른 메시지 처리를 차단하지 않음
- **쿨다운 제어**: 전역 5회 연속 실패 시 자동 대기, `/resume`으로 즉시 해제

### 4. 도구 사용 — 실제 개발 환경 제어

단순한 텍스트 생성이 아니라, Claude Code CLI를 통해 사용자 PC의 개발 환경을 직접 제어합니다:

- 파일 시스템 읽기/쓰기
- bash 명령어 실행
- 코드 수정, 테스트 실행, 린트 적용
- GSD 프레임워크의 슬래시 명령어 호출 (`/gsd:do`, `/gsd:next` 등)
- git 커밋

AI가 에러 로그를 직접 읽고 원인을 분석하여 수정 코드를 적용하는, 완전한 Action-taking 구조입니다.

### 5. 문맥 최적화 — Two-Track 라우팅

비용과 품질의 균형을 위해 메시지를 자동 분류합니다:

```
메시지 수신
  │
  ├─ 경량 모델(Haiku)로 분류 ─┬─ "simple" → 가볍게 처리 (구독 한도 절약)
  │                            └─ "gsd"    → GSD 프레임워크 풀파워 실행
  │
  └─ 분류 호출은 --no-session-persistence (본 세션 오염 방지)
```

세션 관리도 환각(Hallucination) 방지를 위해 설계되었습니다:
- 서브 태스크별 세션 격리 (Fresh Context)
- `max_session_turns` 초과 시 자동 리셋
- GSD 세션 타임아웃(30분) 후 새 세션으로 전환

## 일반 AI 챗봇과의 비교

| 비교 항목 | 일반 메신저 AI 봇 | GSD Orchestrator |
|------|------|------|
| **역할** | 텍스트 생성, 코드 스니펫 제안 | 소프트웨어 개발 전 과정 자율 실행 |
| **실행 모델** | 동기식 (응답까지 대기) | 비동기 큐 (명령 후 일상 복귀) |
| **작업 깊이** | 단일 스텝 문답 | 멀티 스텝 루프 (기획 → 구현 → 검증 → 커밋) |
| **오류 대응** | 사용자가 에러를 복사해서 전달 | AI가 터미널 출력을 직접 읽고 수정 |
| **장애 복구** | 세션 종료 시 모든 맥락 소실 | 파일 기반 상태 보존, 재시작 시 100% 재개 |
| **문맥 관리** | 대화 누적에 따른 환각 증가 | 세션 격리 + 작업 큐 분할로 문맥 오염 방지 |
| **비용 구조** | 토큰 종량제 (사용량 비례 과금) | 구독 플랜 한도를 최적 활용 |
| **채널** | 단일 플랫폼 | Telegram + Slack 동시 지원, 크로스채널 브로드캐스트 |

## 아키텍처 요약

```
[Telegram/Slack] → ChannelAdapter → inbox/{ts}.json
                                         │
                    inbox_processor (10초 폴링) ← 경량 모델 분류
                         │
                    Claude Code CLI + GSD Framework (자율 코딩 또는 즉답)
                         │
                    outbox/{ts}.json (멀티채널 targets)
                         │
                    outbox_sender (3초 폴링) → 채널별 발송 → sent/
                         │
                    archiver (매시간) → archive/ → 만료 삭제
```

단일 Python 프로세스. DB 없음. 외부 서비스 의존 없음.
